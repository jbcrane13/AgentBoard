Architecture — Unified state

Proposing we split AppState into domain-scoped observable stores:

BoardStore       // beads, filters, selection
SessionStore     // tmux sessions, capture, lifecycle
ChatStore        // messages, streaming, session-key
CanvasStore      // history stack, current surface
ConnectionStore  // gateway state, errors, retry
LayoutStore      // sidebar/chat widths, pane visibility
NotificationStore// unread counts, transient alerts

Why

Each store is a small @Observable that views subscribe to selectively. A chat message no longer invalidates the board. Tests instantiate only the store under test.

Dataflow

view ──reads──▶ store
   │
   └──intent──▶ coordinator ──▶ service(s) ──▶ store.mutate

1

One unified top bar replaces three separate nav systems

The v1 app had sidebar nav + center tab bar + right-panel mode picker covering overlapping ground. v2 uses a single breadcrumb (AgentBoard › Project › Destination) that updates based on what's selected in the sidebar or the surface tab strip. Destinations are first-class; modes are gone.

2

Canonical connection indicator — one source of truth

v1 rendered connection state in 3 places with inconsistent styling. v2 shows it once, here, as a persistent chip. Click for details (retry, gateway URL, last error). Everything else binds to the same ConnectionStore.

3

Projects: first-class, sorted, quick-switched

Active project marked with an accent rail + pill. Counts on the right. ⌘1–⌘9 quick-switches. No project shows empty state CTA instead of hiding the section.

4

Sessions: visible state, recoverable errors

Each session shows status dot (running/idle/stopped/error), name, elapsed. Errors surface with a "Reconnect tmux" affordance inline rather than silently disappearing. Clicking a session opens a surface tab (see 5), not a replacement view.

5

Surface tab strip — destinations as tabs

The center panel is a tabbed workspace. Board is the default persistent tab. Opening a session, an epic, or a bead-detail opens a new tab you can close. This replaces the v1 model where Terminal replaced the whole center panel with no breadcrumb back.

6

Project header: scannable, hierarchical

Title + path on the left. Four large, color-coded stat tiles on the right (Open / In Prog / Blocked / Done). Numbers are the hero — not 12px metadata. Clicking a tile scopes the board to that column.

7

Filters as chips — active state is visible

v1 used dropdowns that gave no feedback when active. v2 uses pill-shaped chips that switch to accent when active, with an inline ✕ to clear. Inactive chips are muted. A single "Clear all" appears when ≥2 filters are active.

8

Cards: consistent radii, clear selection, agent visibility

One corner radius (8px). Selection = accent border + soft glow. Assignee avatars bottom-left; git/commit metadata bottom-right in monospace. If an agent is actively working the bead, a pulsing dot replaces the avatar. Hover lifts 1px with softer shadow.

9

Canvas is a drawer, not a mode

Canvas slides in between the center panel and chat. The agent "sending to canvas" animates the drawer open. Chat stays put — you keep typing while reading. Toggle with ⌘.. No mode-picker in the chat header. Canvas has its own history + back/forward, independent of chat scroll.

10

Chat header: minimal, session-first

Session picker + thinking-level chip + new/history. Connection indicator lives in the top bar (#2), not duplicated here. Focus is on the conversation, not on controls.

11

Context bar — chips that reflect the actual prompt envelope

These are the pieces of context attached to the next message (project, active destination, selected bead, gateway session). Removable with ✕. Agent sees exactly what's shown here — no hidden context injection.
